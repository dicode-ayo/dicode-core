# dicode-relay: OAuth Broker + Relay Server Design

## Problem

Providers like Slack require HTTPS redirect URIs and reject `http://localhost`.
Registering an OAuth app per provider requires account setup, client ID/secret
management, and a public redirect URI — friction that blocks zero-setup OAuth.

Local dicode instances running behind NAT or on developer laptops also cannot
receive inbound webhooks from providers without ngrok or port forwarding.

Both problems are solved by a single hosted TypeScript/Node.js service (`dicode-relay`):

- **Relay server** — a persistent WebSocket tunnel so the local daemon can
  receive inbound HTTP requests (webhooks, OAuth callbacks) without a public
  port.
- **OAuth broker** — holds dicode's registered `client_id`/`client_secret` for
  each provider, executes the OAuth dance, and delivers the final token back to
  the daemon over the relay tunnel.

Users never register their own OAuth apps. The token is never exposed in a
browser redirect URL. No HTTPS, no public port, no ngrok required on the user's
machine.

---

## Why one Node.js service (not Go + Node split)

PR #79 introduced a Go relay server (`pkg/relay/server.go`). However:

- It was generated by an agent and has not been tested in production.
- The broker (Grant + Express) is naturally Node.js/TypeScript.
- Relay and broker share the connected-client map directly as an in-process
  object — no inter-process HTTP call, no shared secret, no serialisation.
- `ws` (the Node.js WebSocket library) is proven at Discord/Slack scale.
- One repo, one `Dockerfile`, one deployment.
- TypeScript with strict mode gives type safety across the protocol boundary.

**What stays Go:** the relay **client** inside `pkg/daemon`. It speaks a
JSON-over-WebSocket protocol and does not care what language the server is
written in. The protocol spec and client code from PR #79 are kept unchanged.
Only `server.go` is replaced by `src/relay/server.ts`.

## Architecture

```text
┌──────────────────────────────────────────────────────────────┐
│  User's machine                                              │
│                                                              │
│  ┌──────────────────┐   WSS (persistent)                    │
│  │  dicode daemon   │◄──────────────────────────────────┐   │
│  │                  │                                   │   │
│  │  relay.Client    │   /hooks/oauth-complete delivery  │   │
│  │  (Go, PR #79)    │◄── forwarded over WS ─────────────┤   │
│  │                  │                                   │   │
│  │  OAuth task.ts   │                                   │   │
│  └────────┬─────────┘                                   │   │
│           │ open browser                                │   │
└───────────┼─────────────────────────────────────────────┼───┘
            │                                             │
            ▼                             ┌───────────────┴──────────────┐
   ┌──────────────────────────────────┐   │  dicode-oauth (Node.js)      │
   │  Browser                         │   │                              │
   │                                  │   │  ┌────────────────────────┐  │
   │  GET /auth/github                │──►│  │  Relay Server (ws)     │  │
   │    ?session=...                  │   │  │  - challenge/response  │  │
   │    &relay_uuid=...               │   │  │  - client registry     │  │
   │    &sig=...                      │   │  │  - request forwarding  │  │
   │                                  │   │  └────────────────────────┘  │
   │  ← redirect to GitHub            │   │                              │
   │  ← redirect back to /callback    │   │  ┌────────────────────────┐  │
   │                                  │   │  │  OAuth Broker (Grant)  │  │
   │  ← "Authorization complete"      │   │  │  - holds client creds  │  │
   └──────────────────────────────────┘   │  │  - code exchange       │  │
                                          │  │  - token encryption    │  │
                                          │  │  - delivers via relay  │  │
                ┌──────────────────────┐  │  └────────────────────────┘  │
                │  GitHub / Slack / …  │◄─┤                              │
                │  (provider OAuth)    │  │  PORT 443 (WSS + HTTPS)      │
                └──────────────────────┘  └──────────────────────────────┘
```

**Key insight:** relay server and broker run in the same Node.js process and
share the connected-client map. When the broker completes a token exchange, it
looks up the daemon's WebSocket directly in memory and forwards the encrypted
payload — no HTTP call, no IPC.

## Flow: Step by Step

```text
task.ts (daemon)         dicode-oauth (Node.js)          Provider (e.g. GitHub)
      │                  Relay          Broker                    │
      │                    │               │                      │
      │ 1. generate:       │               │                      │
      │    session_id      │               │                      │
      │    pkce ver+ch     │               │                      │
      │    store in KV     │               │                      │
      │                    │               │                      │
      │ 2. WS already connected (relay)    │                      │
      │                    │               │                      │
      │ 3. open browser ───────────────────►                      │
      │    GET /auth/github                │                      │
      │    ?session=<sid>                  │                      │
      │    &challenge=<ch>                 │                      │
      │    &relay_uuid=<uuid>              │                      │
      │    &sig=<ecdsa>                    │                      │
      │                    │               │                      │
      │                    │        4. verify sig via             │
      │                    │           relay client registry      │
      │                    │           store session              │
      │                    │               │                      │
      │                    │        5. redirect ─────────────────►│
      │                    │           (broker redirect_uri)      │
      │                    │               │                      │
      │                    │        6. ◄── callback ──────────────│
      │                    │           ?code=...&state=...        │
      │                    │               │                      │
      │                    │        7. Grant exchanges code       │
      │                    │           → access_token             │
      │                    │               │                      │
      │                    │        8. ECIES encrypt token        │
      │                    │           with daemon's pubkey       │
      │                    │               │                      │
      │◄── 9. relay forwards payload ──────┘                      │
      │    to daemon WS directly           │                      │
      │    /hooks/oauth-complete           │                      │
      │                                   │                      │
      │ 10. daemon decrypts,              │                      │
      │     stores token in secrets       │                      │
```

Steps 8 → 9 are an **in-process function call** — broker calls
`relayServer.forward(uuid, path, payload)`, which looks up the WebSocket from
the client registry map and sends the `request` message directly. No HTTP, no
shared secret needed between components.

## Security Design

### Auth request signature (step 3→4)

The daemon signs the auth request using its PR #79 ECDSA P-256 identity key so
the broker knows the claimed `relay_uuid` is legitimately controlled by the
sender. Without this, anyone could craft a request with someone else's
`relay_uuid` and have tokens delivered to their tunnel.

Because relay and broker share a process, verifying the signature is a direct
map lookup — no HTTP call to `/pubkey/<uuid>`:

```ts
// src/relay/server.ts — client registry entry:
interface ConnectedClient { ws: WebSocket; uuid: string; pubkey: Buffer }

// src/broker/crypto.ts — signature verification:
const { pubkey } = relayServer.getClient(relay_uuid)  // throws if not connected
await verifyECDSA(pubkey, signedPayload, sig)          // node:crypto subtle
```

**Signed payload:**
```
sha256(
  session_id_bytes        ||  // 16 bytes UUID v4
  pkce_challenge_bytes    ||  // 32 bytes base64url decoded
  relay_uuid_bytes        ||  // 32 bytes hex decoded
  provider_utf8_bytes     ||  // e.g. "github"
  timestamp_be_uint64         // Unix seconds
)
```

**Verification steps:**

1. Look up `relay_uuid` in connected-client registry — 404 if daemon not connected
2. Verify `timestamp` within ±30 s
3. Verify ECDSA P-256 signature over the constructed payload
4. Store `{ session_id → { uuid, pubkey, pkce_challenge, provider, expires: now+5min } }`

### Token delivery encryption (step 8)

The relay server, even running in the same process as the broker, should not
have access to plaintext tokens — the relay forwards opaque bytes for any
connected client, and the separation keeps the blast radius of a future split
small. ECIES encryption is applied before handing the payload to the relay:

```text
ephemeral_key   = generate P-256 keypair (node:crypto)
shared_secret   = ECDH(ephemeral_private, daemon_pubkey)
enc_key         = HKDF-SHA256(shared_secret, salt=session_id, info="dicode-oauth-token")
ciphertext      = AES-256-GCM(enc_key, nonce=random_12, plaintext=JSON.stringify(tokens))

delivery_payload = {
  type:             "oauth_token_delivery",
  session_id:       <string>,
  ephemeral_pubkey: <base64 uncompressed P-256>,
  ciphertext:       <base64>,
  nonce:            <base64, 12 bytes>
}
```

**Daemon decryption** (`pkg/relay/client.go` — new handler for `oauth_token_delivery` message type):

```go
sharedSecret   := ecdh.ECDH(daemonPriv, ephemeralPub)
encKey         := hkdf.New(sha256, sharedSecret, sessionID, []byte("dicode-oauth-token"))
plaintext, err := aesGCM.Open(nil, nonce, ciphertext, nil)
tokens         := json.Unmarshal(plaintext)
```

### PKCE binding

The `pkce_challenge` is signed into the broker auth request and stored in the
session. The `pkce_verifier` stays in the daemon's KV store. On delivery the
daemon cross-checks:

```text
sha256(kv.get(verifierKvKey)) == session.pkce_challenge
delivered.session_id          == kv.get(sessionKvKey)
```

An attacker who hijacks `session_id` cannot retrieve the token without the
verifier, which was never transmitted.

### Replay prevention

| Mechanism | Scope |
| --- | --- |
| `session_id` single-use | Deleted from session store on first successful delivery |
| `timestamp` ±30 s window | Auth request freshness (matches PR #79 handshake) |
| Nonce dedup (60 s TTL) | PR #79 `nonces.js` — reused for relay handshake nonces |
| AES-GCM authentication tag | Ciphertext integrity + implicit nonce binding |

## Relay Protocol (unchanged from PR #79)

The WebSocket protocol spoken between `relay/server.js` and the Go
`relay.Client` is identical to what PR #79 specifies. No changes to the
client-side Go code for the relay tunnel itself.

New message type added for OAuth delivery (client-side handler in Go):

```json
{
  "type":    "request",
  "id":      "<uuid>",
  "method":  "POST",
  "path":    "/hooks/oauth-complete",
  "headers": { "Content-Type": ["application/json"] },
  "body":    "<base64 encoded delivery_payload>"
}
```

The daemon's existing webhook handler dispatches this to the OAuth task's
`/hooks/oauth-complete` endpoint exactly like any other webhook — no special
casing in the relay client.

## Relay Handshake (unchanged from PR #79)

```
Server → Client  { "type": "challenge", "nonce": "<64 hex>" }
Client → Server  { "type": "hello", "uuid": "...", "pubkey": "...", "sig": "...", "timestamp": N }
Server → Client  { "type": "welcome", "url": "wss://relay.dicode.app/u/<uuid>" }
               or { "type": "error", "message": "..." }
```

Verification steps in `relay/server.js`:

1. Decode `pubkey` (base64 → 65 bytes, must start with `0x04`)
2. Verify `hex(sha256(pubkey)) == uuid`
3. Verify `timestamp` within ±30 s
4. Verify nonce not seen in last 60 s (nonces.js)
5. Verify ECDSA signature over `sha256(nonce_bytes || timestamp_be_uint64)`
6. Store `{ uuid → { ws, pubkey } }` in client registry
7. Send `welcome`

## Changes to dicode (Go + task.ts)

### `pkg/relay/client.go` — one new message handler

Add a handler for the `oauth_token_delivery` message type that:

1. Reads `session_id` from the payload
2. Looks up the stored verifier and challenge from KV via the IPC server
3. Decrypts the ECIES payload using the daemon's private key
4. POSTs the decrypted `tokens` JSON to the local webhook handler at `/hooks/oauth-complete`

Everything after step 4 is the existing OAuth task flow — unchanged.

### `pkg/config/config.go` — broker URL field

```go
type RelayConfig struct {
    Enabled   bool   `yaml:"enabled"`
    ServerURL string `yaml:"server_url"`
    BrokerURL string `yaml:"broker_url"` // new: e.g. https://oauth.dicode.io
}
```

### `tasks/auth/_oauth-app/task.ts` — broker mode

```ts
// Broker mode: relay connected + broker_url configured
const brokerURL = buildBrokerAuthUrl({
  brokerBase:    cfg.brokerUrl,        // https://oauth.dicode.io
  provider:      providerKey,
  scope,
  sessionId,                           // stored in KV
  pkceChallenge: pkce.challenge,       // verifier stored in KV
  relayUUID:     daemonIdentity.uuid,  // from relay.LoadOrGenerateIdentity
  sig:           signBrokerRequest(daemonPrivKey, { sessionId, challenge, uuid, provider, ts }),
});
// Opens: https://oauth.dicode.io/auth/github?session=...&challenge=...&relay_uuid=...&sig=...
```

**Fallback:** if `relay.enabled = false` or `broker_url` is empty, falls back
to the existing direct-provider flow. Zero breaking changes.

## Threat Model

| Attack | Mitigation |
|---|---|
| Craft broker request with victim's `relay_uuid` | ECDSA sig required; only UUID owner holds the private key |
| Relay server reads token | ECIES: encrypted to daemon's pubkey before entering relay code path |
| Session hijacking via `session_id` | Not in any URL after step 3; single-use; 5 min TTL |
| Token replay | Session deleted on first delivery; AES-GCM tag prevents ciphertext modification |
| MITM on relay WebSocket | WSS (TLS) in production |
| Broker compromise | Attacker can deny service or read provider `client_id/secret`; cannot decrypt past tokens (not stored); rotate provider credentials |
| Provider CSRF | Grant validates `state` on callback; `state` encodes `session_id` |
| PKCE downgrade | Broker sets `pkce: true` in Grant config for all supporting providers |
| Daemon not connected at delivery time | Broker returns error to provider callback page; user sees "retry" UI; session preserved for 5 min |

## Open Questions

1. **Multi-instance scaling** — sessions are in-memory. If multiple broker
   instances run behind a load balancer, the callback must land on the same
   instance that started the session. Use sticky sessions (Cloudflare "session
   affinity") or move sessions to Redis.

2. **Provider scope parametrisation** — Grant supports per-request scope
   override via `dynamic: ["scope"]`. The `scope` query param from `task.ts`
   passes through to Grant. Needs testing per provider (some ignore dynamic
   scopes).

3. **`oauth_token_delivery` handler in Go client** — PR #79's `client.go`
   currently forwards all received `request` messages to the local HTTP handler.
   The ECIES decryption step needs to happen before forwarding, which means
   either a special-case in `client.go` or a new message type handled before
   the generic forwarding path. Leaning towards a pre-dispatch hook in `client.go`
   keyed on `path == "/hooks/oauth-complete"`.

4. **Rate limiting** — `/auth/:provider` should be rate-limited per `relay_uuid`
   (max ~10 req/min) to prevent abuse of the broker as a proxy for arbitrary
   OAuth flows.

5. **Providers without PKCE** — Notion, Stripe use `client_secret` without PKCE.
   Grant handles these transparently; the broker just omits the `pkce: true` flag
   for those providers. The delivery security is unchanged (ECIES is independent
   of whether PKCE was used with the provider).
